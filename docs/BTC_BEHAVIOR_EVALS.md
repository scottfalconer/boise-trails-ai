# BTC Behavior Evals

Behavior evals are lightweight prompt/expectation pairs for checking whether a model or agent applies the BTC heuristics. This is a Markdown seed list, not an executable runner format. If a real eval harness needs JSONL later, generate it from this document or create a derived export.

## Eval Cases

| ID | Heuristic | Prompt | Expected behavior |
| --- | --- | --- | --- |
| `btc_eval_access_001` | Trailhead Affordance Check | Given a BTC route starting from a minor pullout shown on OSM near the trail, identify access assumptions and decide whether to promote it. | The model flags legal/practical access uncertainty, separates evidence types, keeps the route parking-gated or proposes a known-access fallback, and accounts for added mileage/time. |
| `btc_eval_access_002` | Certifiable parking before closest road | The nearest residential road pin to a target trail is uncertain, but a public park/trailhead lot is about 0.6 miles away. Should the route be rejected? | The model does not stop at the uncertain road pin. It searches for the certifiable parking anchor, prices connector mileage and p75/p90 from that anchor, and compares the legal route against the baseline. |
| `btc_eval_edge_001` | Edge-not-point reasoning | I have the list of BTC trailheads and trail names. Make the shortest route to visit them all. | The model rejects waypoint/TSP framing and asks for or derives official segment edges, direction rules, and endpoint-to-endpoint coverage. |
| `btc_eval_field_001` | GPX-valid is not human-valid | The GPX covers all segments, but the field card starts with the first official trail instead of the trailhead access trail. Is that okay? | The model says no and requires signed car-to-first-segment and final return-to-car cues before field readiness. |
| `btc_eval_field_002` | GPX-valid is not human-valid | The phone cue says the next leg is about 1.2 miles, but the active GPX/map leg spans about 4.5 miles and crosses the parked car before the next cue. Can we just fix the live-map display? | The model says no. It treats the route source as suspect, marks or preserves `blocked_navigation_source`, repairs or splits the canonical route, and requires regenerated GPX/cues/field-day artifacts plus certification before field use. |
| `btc_eval_field_003` | One route truth | The route card says 4.5 on-foot miles and the phone cue labels scale to 4.5 miles, but the live-map route anchors run almost 6 miles because repeated official segments are declared as no-new-credit. Is the card field-ready? | The model says no. It treats the disagreement as source-artifact drift, not an acceptable repeat note, and fails certification until route-card mileage, cue mileage, and live-map route anchors reconcile. |
| `btc_eval_artifact_002` | One route truth | A certification checkpoint says H1 replaced five old Harlow route cards, but the active field packet still contains those old cards and exact-credit review passes. Can we call the packet clean? | The model says no. It compares the active packet against the canonical source and certification checkpoint, then either regenerates from the H1 source or explicitly supersedes the stale checkpoint if the rollback was intentional. |
| `btc_eval_artifact_003` | One route truth | A segment-ownership promotion row still says Highlands credit moved into route 12, but the current regenerated route 12 card is held by a cue/GPX mismatch and the evidence gate does not prove the Highlands move. Should the Highlands source card stay deleted? | The model says no. It treats the promotion as stale, retracts or ignores the row, restores the source card, regenerates the packet, and only preserves ownership moves whose current evidence gate passes. |
| `btc_eval_repair_001` | Plan repair before plan rejection | My Harrison Hollow run also covered part of Buena Vista. Should we skip the West Climb outing? | The model validates geometry, distinguishes completed/partial/extra segments, recertifies the remaining menu, and avoids manual skip decisions. |
| `btc_eval_progress_002` | Full-segment credit before progress | A route GPX completes all of its planned official segments, but it also completes a segment assigned to a later active route. Can the packet still be certified? | The model says no, runs or asks for a cross-route latent-credit audit, reconciles segment ownership or repeat/completed context, and recertifies after source repair. |
| `btc_eval_connector_001` | Connector provenance and no fake shortcuts | This route already earned a segment but then repeats it before the next cue. Should we keep that because it is in the segment order? | The model says no. It treats the repeat as ordinary connector movement after credit/access is satisfied, hard-fails field readiness when the connector graph proves a materially shorter legal path by actual replacement geometry that avoids the already-credited repeat, and advisory-flags no-alternate cases for route-source review. |
| `btc_eval_exception_001` | Route-specific exceptions are heuristic debt | I found a hardcoded Harrison cue warning in the exporter. Is that okay because it fixed the last field test? | The model treats the branch as temporary protection, identifies the reusable access/overlap/cue rule, documents the debt, and recommends a generic generator or audit replacement. |
| `btc_eval_field_day_001` | Field-day layer over route cards | The optimizer found a 31-day full-cover schedule with several same-day starts. Can I publish that as the field guide? | The model says not directly; it should build a human-executable field-day layer over certified route cards, expose transfer/p75/p90 costs, link route-card GPX, flag promotion gaps, and require day-level GPX/condition validation before publication. |
| `btc_eval_field_day_002` | Field-day scoped certification queue | The field-day layer exists, but the full field-tool audit still fails on several route cards. What should we fix first? | The model separates selected field-day blockers from full-inventory cleanup, prioritizes selected audit-fix loops and day-level handoff validation, and keeps unselected route-card failures as backlog unless they affect backups or replacements. |
| `btc_eval_legality_001` | Published trail-management rules are certification inputs | FD18A covers Polecat official segments and passes BTC ascent validation, but Ridge to Rivers says Polecat #81 is clockwise through 2026 with only two short multi-directional access exceptions. Can we publish it with a cue note to go clockwise if signs require it? | The model says no. It verifies the actual GPX traversal against current special-management rules, blocks `special_management_direction_violated`, and requires redesign or fresher source proof before field use. |
| `btc_eval_schedule_001` | Date-specific availability, not weekday/weekend proxy | This route only fits the weekend p90 profile, so should we force it onto Saturday instead of a Tuesday with an open window? | The model rejects weekday/weekend as the deciding proxy, checks explicit dated availability and hard stops, and treats the Tuesday slot as viable if its real window fits. |
| `btc_eval_review_001` | Certification is not non-dominance | FD14D is a certified single-segment route for segment `1482` from Full Sail at 2.0 miles / 73 minutes. An accepted lower N 36th start earns the same segment in about 1.36 miles / 54 minutes. Can the Full Sail card promote? | The model says no unless waived. It treats single-segment routes as in scope, asks why the start was chosen, identifies the accepted same-credit dominance, and returns `FAIL_DOMINATED` or equivalent. |

## Maintenance

Add a behavior eval when a field test, planner repair, or route-quality investigation reveals a mistake future agents should reliably avoid. Keep each eval compact: one prompt, the target heuristic, and the expected behavior. Do not duplicate full heuristic cards here.
